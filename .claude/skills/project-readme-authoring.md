# 项目 README 编写技能（project-readme-authoring）

## 技能定位
本技能用于输出**标准化、专业化、高可读性**的项目 `README.md` 文档，严格遵循 GitHub/Gitee 开源社区通用规范，适配求职面试、开源交付、课程毕设、团队协作等多场景。重点突出项目价值、技术栈、可复现性与技术亮点，可直接用于面试官评审与项目对外交付。

## 适用场景
- 求职面试：个人项目仓库 README 包装，突出个人能力与项目成果
- 开源项目：官方项目说明文档，符合社区通用规范
- 课程/毕设：课程作业、毕业设计项目说明文档
- 团队交付：项目交接文档，保障后续人员可快速上手

## 核心能力
1. **标准结构搭建**：遵循行业通用的 10 模块经典结构，自动生成锚点目录，标题层级严谨，逻辑递进清晰
2. **多类型适配**：支持前后端分离、纯前端、纯后端、算法/大数据、小程序、毕设等不同项目类型，自动调整内容侧重点
3. **专业排版美化**：内置技术徽章、数据表格、多语言代码高亮、居中图文、提示引用块等排版能力，视觉统一官方
4. **面试向优化**：通过引导式提问提炼技术亮点、量化性能成果、明确个人职责、规避口语化表达，强化工程素养呈现
5. **可复现保障**：标准化部署启动流程，明确环境版本、依赖安装、配置修改、运行命令，确保他人可一键复现项目

## 工作流程
当用户需要生成项目 README 时，按以下步骤执行：

### 第一步：信息确认
- 确认项目类型（前后端分离/纯前端/纯后端/算法/小程序/毕设等）
- 确认核心技术栈与对应版本
- 确认使用场景（面试/开源/毕设/团队交付）
- 确认项目核心业务与个人负责模块

**引导式提问**（用于提炼技术亮点）：
- 项目遇到过哪些性能瓶颈？最终如何解决？有无优化前后数据对比？
- 是否设计了可复用的公共组件/工具库/算法封装？
- 做过哪些安全性加固（XSS/SQL注入/权限/加密）？具体方案是什么？
- 系统上线后支撑了多少用户？最大并发/QPS 是多少？
- 团队协作中你负责了哪些核心模块？主导了哪些技术选型？

### 第二步：模板匹配
根据项目类型与场景，从核心模板库中选择对应基础框架。

### 第三步：内容生成
按模块依次生成完整文档，重点保障：
- **项目简介**：一句话定位 + 业务背景 + 项目规模 + 个人职责（多种话术可选）
- **技术栈**：分层罗列，核心技术用行内代码标注版本，统一格式（如 `Vue 3.3`）
- **部署步骤**：分端编写，命令完整可直接复制执行，消除任何格式混乱
- **技术亮点**：量化成果，体现技术深度与个人贡献，避免空泛描述

### 第四步：排版优化
- 添加技术徽章（shields.io，链接指向官网或仓库）
- 表格对齐、代码块标注对应语言（json/yaml/properties 等也需高亮）
- 统一图标与分割线，合理留白分层
- 图片统一存放在 `docs/images/`，使用相对路径，单张压缩至 200KB 内

### 第五步：质量校验
对照质量校验清单逐项检查，输出最终可直接使用的 README 内容。

<h2 id="3-architecture">3. 系统架构</h2>

采用前后端分离分层架构，自上而下分为：
1. 表现层：前端页面交互、数据可视化、表单校验

<h2 id="8-highlights">8. 技术亮点</h2>
（以下为示例，需根据实际项目替换）

1. **缓存性能优化**：热点数据接入 Redis，采用缓存预热 + 过期时间随机化方案，解决缓存雪崩，核心接口响应耗时从 220ms 降至 28ms
2. **权限架构设计**：基于 RBAC 实现动态权限，无硬编码菜单，支持按钮级粒度控制，灵活适配不同角色
3. **大文件上传优化**：分片上传 + 断点续传，解决 GB 级文件上传超时，支持进度实时展示
4. **SQL 查询优化**：针对慢查询添加联合索引，采用延迟分页优化深分页，百万级数据分页查询耗时 <100ms
5. **全局工程规范**：统一返回结果封装，全局异常拦截，自定义业务状态码，前端无需重复判断接口状态
6. **容器化部署**：编写 Dockerfile 与 docker-compose.yml，实现一键部署整套服务，降低环境搭建成本

## 质量校验清单
生成完成后必须对照检查，确保文档质量：

- ✅ **结构规范**：标题层级无跳级，一级标题仅1个，目录锚点可正常跳转（推荐使用 HTML 锚点避免中文 id 兼容问题）
- ✅ **核心完整**：包含项目简介、技术栈、启动步骤、技术亮点四大必选模块
- ✅ **排版统一**：代码块标注对应语言（包括 json/yaml/properties），表格对齐，图片有说明，分割线分隔大模块
- ✅ **面试适配**：有量化成果数据，个人职责明确（根据场景选择话术），无口语化、情绪化表述
- ✅ **可复现**：部署步骤完整，依赖版本明确，数据库脚本含测试数据，无缺失关键配置说明
- ✅ **术语统一**：技术名词大小写规范，版本号写法统一（如 `Vue 3.3`），无前后矛盾内容
- ✅ **图片安全**：使用相对路径存放在项目内（如 `docs/images/`），无外链图片或绝对路径，已压缩

## 进阶优化技巧
1. **技术徽章**：使用 [shields.io](https://shields.io) 生成自定义徽章，徽章链接指向官方文档或仓库首页，提升专业感与可交互性
2. **配图规范**：首页放置1-2张核心截图+架构图，全部存放于 `docs/images/` 下，使用相对路径引用，图片需压缩至 200KB 以内，建议使用 WebP 格式
3. **开源协议**：末尾添加 MIT/Apache 2.0 等开源协议，符合社区规范
4. **更新日志**：大型项目可新增 `CHANGELOG.md` 并在 README 中链接，记录版本迭代内容
5. **贡献指南**：开源项目可新增 `CONTRIBUTING.md`，说明代码提交规范与 Issue 模板
6. **在线演示**：如有部署环境，在"效果展示"附近提供演示地址和测试账号，大幅提升面试/开源体验

## 避坑红线
绝对避免以下问题：
- ❌ 大段纯文字无分层，无目录无标题，可读性极差
- ❌ 只罗列业务功能，无技术亮点与优化思考
- ❌ 口语化表述，如"随便写的""很好用""踩了很多坑"
- ❌ 缺失完整启动步骤，他人无法复现项目（命令混入无关文字、代码块未闭合）
- ❌ 图片使用绝对路径或外链，导致离线无法查看
- ❌ 夸大成果但无数据支撑，前后内容矛盾
- ❌ 锚点因中文编码失效，目录点击无法跳转
